# Ключевые механики

## 1. Терминальный интерфейс

### Описание
Командная строка с интеллектуальным парсером ввода. Все команды абстрагированы от реальных уязвимостей, но опираются на реальные принципы работы сетей и систем безопасности.

### Базовые команды

| Команда | Описание | Параметры | Пример |
|---------|----------|-----------|--------|
| `scan` | Сканирование сети или узла | `--ports`, `--services`, `--os` | `scan 192.168.1.0/24 --ports` |
| `route` | Построение маршрута к цели | `--via`, `--proxy` | `route 10.0.0.5 --via 192.168.1.1` |
| `connect` | Подключение к узлу | `--port`, `--protocol` | `connect 10.0.0.5 --port 22` |
| `decrypt` | Дешифровка данных | `--algorithm`, `--key` | `decrypt file.enc --algorithm RSA` |
| `patch` | Применение исправлений | `--vuln-id`, `--target` | `patch CVE-2001-0001 --target server1` |
| `monitor` | Мониторинг трафика | `--filter`, `--duration` | `monitor --filter "port 80"` |
| `extract` | Извлечение данных | `--source`, `--dest` | `extract --source db --dest usb` |
| `bypass` | Обход защиты | `--method`, `--target` | `bypass firewall --method tunnel` |
| `help` | Справка по командам | `[command]` | `help scan` |

### Геймификация терминала

**Встроенный словарь команд:**
- Контекстные подсказки при вводе
- История последних 100 команд
- Избранные команды для быстрого доступа

**Режим обучения:**
- Пошаговые туториалы для каждой команды
- Безопасная песочница для экспериментов
- Проверка синтаксиса в реальном времени

**Достижения за использование терминала:**
- «Первый скан» — первое успешное сканирование
- «Полиглот» — использование 10 различных команд
- «Мастер скриптов» — создание сложного маршрута из 5+ узлов

## 2. Сетевая топология

### Описание
Визуальная карта узлов, портов и уровней доступа. Игрок строит маршруты, обходит фаерволы, настраивает прокси, перехватывает пакеты.

### Структура сети

```
                    [INTERNET]
                        │
                   [ROUTER]
                  /    │    \
           [FIREWALL]  │     [DMZ]
                │      │       │
          [INTERNAL]───┘    [WEB SERVER]
           /    │    \
      [DB]   [APP]   [WORKSTATIONS]
```

### Типы узлов

| Тип узла | Описание | Требования | Риск обнаружения |
|----------|----------|------------|------------------|
| **Рабочая станция** | Обычный компьютер сотрудника | Минимальные | Низкий |
| **Сервер приложений** | Бизнес-логика компании | Средние | Средний |
| **База данных** | Хранение критических данных | Высокие | Высокий |
| **Фаервол** | Защита периметра сети | Специфические | Очень высокий |
| **Маршрутизатор** | Управление трафиком | Продвинутые | Высокий |
| **DMZ-зона** | Публичные сервисы | Средние | Средний |

### Головоломка на графах

**Механика:**
- Каждый узел представлен как вершина графа
- Соединения между узлами — рёбра с параметрами
- Игрок должен найти оптимальный путь от входа к цели

**Параметры узла:**
- **Требования к ресурсам:** Сколько вычислительных циклов нужно
- **Вес обнаружения:** Насколько вырастет уровень тревоги
- **Время прохождения:** Сколько игрового времени займёт
- **Необходимые навыки:** Какие перки требуются

**Пример головоломки:**
```
Старт → A(3 цикла, +10% риск) → B(5 циклов, +20% риск) → Цель
           ↓                        ↑
        C(2 цикла, +5% риск) ───────┘

Задача: Выбрать маршрут A→B или C→B с учётом лимита циклов и допустимого риска
```

**Ошибки:**
- Неправильный выбор пути → таймер обратного отсчёта
- Превышение лимита циклов → блокировка узла на 30 секунд
- Критический уровень обнаружения → провал миссии

## 3. Физическое взаимодействие

### Описание
Мини-игры на точность и время для работы с физическим оборудованием в серверной.

### Типы взаимодействий

#### A. Замена накопителей

**Сценарий:** Нужно заменить ленточный накопитель или жёсткий диск

**Механика:**
1. Открыть отсек сервера (клик по защёлкам)
2. Извлечь старый накопитель (удерживать кнопку, тянуть)
3. Вставить новый накопитель (совместить направляющие)
4. Зафиксировать защёлки

**Риски:**
- Статическое электричество (нужен антистатический браслет)
- Неправильная ориентация накопителя
- Повреждение контактов при грубом извлечении

#### B. Переподключение кабелей

**Сценарий:** Нужно изменить схему подключения патч-панели

**Механика:**
1. Выбрать правильный тип кабеля (витая пара, оптоволокно)
2. Подключить к правильному порту (по цветовой маркировке)
3. Уложить кабель в кабель-менеджер
4. Проверить соединение индикаторами

**Риски:**
- Перепутанные пары → отсутствие связи
- Слишком сильный изгиб оптоволокна → повреждение
- Незащёлкнутый коннектор → нестабильное соединение

#### C. Сброс питания

**Сценарий:** Требуется перезагрузить зависший сервер

**Механика:**
1. Найти правильный автомат в щитке
2. Подтвердить действие (предупреждение о потере данных)
3. Переключить автомат в положение OFF
4. Выждать 10 секунд
5. Переключить в положение ON

**Риски:**
- Не тот автомат → отключение критических систем
- Слишком быстрое включение → повреждение оборудования
- Отсутствие резервного питания → потеря данных

#### D. Ручное охлаждение

**Сценарий:** Система охлаждения отказала, температура растёт

**Механика:**
1. Обнаружить перегретые стойки (тепловизор/индикаторы)
2. Подключить мобильный кондиционер
3. Направить воздушный поток на горячие зоны
4. Контролировать температуру в реальном времени

**Риски:**
- Конденсат на оборудовании → короткое замыкание
- Недостаточное охлаждение → аварийное отключение
- Шум мобильного кондиционера → привлечение внимания

#### E. Считывание MAC-адресов

**Сценарий:** Нужно идентифицировать конкретное устройство в сети

**Механика:**
1. Найти наклейку на устройстве (осмотр с близкого расстояния)
2. Сфотографировать или записать адрес
3. Ввести в терминал для привязки к IP

**Формат MAC-адреса:**
```
XX:XX:XX:XX:XX:XX
00:1A:2B:3C:4D:5E
```

**Риски:**
- Плохая видимость (темнота, грязь)
- Ошибка при записи одной цифры
- Привлечение внимания при длительном осмотре

## 4. Режим защиты (Blue Team)

### Описание
Анализ логов, настройка правил фильтрации, изоляция заражённых сегментов, реагирование на инциденты в реальном времени.

### Основные задачи Blue Team

#### Анализ логов

**Интерфейс:**
```
[LOG VIEWER v2.1]
─────────────────────────────────────────
TIME        │ SOURCE      │ EVENT          │ SEVERITY
─────────────────────────────────────────
14:32:15    │ 192.168.1.45│ Failed login   │ MEDIUM
14:32:18    │ 192.168.1.45│ Failed login   │ MEDIUM
14:32:21    │ 192.168.1.45│ Failed login   │ HIGH
14:32:24    │ 192.168.1.45│ Account locked │ CRITICAL
─────────────────────────────────────────
[FILTER] [SEARCH] [EXPORT] [ALERT RULES]
```

**Механика:**
- Поиск аномальных паттернов в потоке событий
- Фильтрация по источнику, типу события, уровню важности
- Корреляция событий из разных источников

#### Настройка правил фильтрации

**Интерфейс:**
```
[FIREWALL RULES EDITOR]
─────────────────────────────────────────
Rule #1: ALLOW TCP port 80 from ANY to WEB_SERVER
Rule #2: ALLOW TCP port 443 from ANY to WEB_SERVER
Rule #3: DENY ALL from EXTERNAL to DATABASE
Rule #4: [NEW RULE...]
─────────────────────────────────────────
[SOURCE] [DESTINATION] [PORT] [PROTOCOL] [ACTION]
```

**Механика:**
- Создание правил с приоритетами
- Тестирование правил на симулированном трафике
- Баланс между безопасностью и доступностью

#### Изоляция заражённых сегментов

**Сценарий:** Обнаружен вирус в сегменте сети

**Механика:**
1. Идентифицировать заражённые хосты
2. Отключить порты на коммутаторе
3. Перенаправить трафик через карантинную зону
4. Применить сигнатуры антивируса

**Риски:**
- Ложная изоляция → простой бизнеса
- Недостаточная изоляция → распространение угрозы
- Паника пользователей → звонки в службу поддержки

### Пазл на приоритизацию

**Условия:**
- Ограниченные ресурсы (администратор один)
- Нарастающая волна атак
- Необходимость сохранения uptime > 99%

**Пример ситуации:**
```
ВРЕМЯ: 14:30
СОБЫТИЯ:
├─ [CRITICAL] DDOS-атака на веб-сервер
├─ [HIGH] Подозрительная активность в базе данных
├─ [MEDIUM] Пользователь сообщает о странном поведении ПК
└─ [LOW] Плановое обновление сертификатов

РЕСУРСЫ: 3 очка действия на ход

ВОПРОС: Что делать в первую очередь?
```

**Правильная стратегия:**
1. Критические угрозы → немедленно
2. Высокие угрозы → в течение 2 ходов
3. Средние угрозы → в течение 5 ходов
4. Низкие приоритеты → когда есть время

**Последствия ошибок:**
- Пропуск критической угрозы → провал миссии
- Неправильная приоритизация → частичный успех с штрафами
- Оптимальное распределение → бонусы к репутации

## 5. Социальная инженерия (дополнительная механика)

### Описание
Взаимодействие с NPC для получения информации, доступа или отвлечения внимания.

### Типы взаимодействий

**Телефонный звонок:**
- Выбор тона разговора (формальный, дружеский, срочный)
- Использование социальной инженерии («Я из IT, нужна помощь»)
- Риск разоблачения при неправильных ответах

**Личный контакт:**
- Выбор места и времени для встречи
- Предложение сделки или шантаж
- Отслеживание реакции NPC

**Фишинг:**
- Создание поддельных писем
- Выбор целевой аудитории
- Мониторинг процента успешных кликов

## 6. Система крафта скриптов

### Описание
Комбинирование базовых команд в сложные скрипты для автоматизации задач.

### Интерфейс редактора скриптов

```
[SCRIPT EDITOR v1.0]
─────────────────────────────────────────
SCRIPT NAME: stealth_scan_v2
AUTHOR: Player
VERSION: 2.0
─────────────────────────────────────────
LINE 01: scan target --stealth --slow
LINE 02: route via proxy_1, proxy_2
LINE 03: connect --encrypted
LINE 04: if detection > 50% then abort
LINE 05: extract --compress --encrypt
─────────────────────────────────────────
[COMPILE] [TEST] [SAVE] [RUN]
```

### Механика
- Drag-and-drop блоков команд
- Установка условий и циклов
- Тестирование в песочнице
- Публикация в сообществе (для ежедневных контрактов)
